01SUBJECT

What it actually is.

Three independent agents converged on the same picture. A small, opinionated, non-custodial Telegram-native gating bot — opinionated in ways that are sometimes elegant, sometimes risky.

TG Token Gates is a non-custodial Telegram bot that gates private groups, channels, and channel+discussion pairs by on-chain holdings across Solana · Ethereum · Base · BNB. Owners pay a one-time SOL fee, configure up to 5 token gates + 1 net-worth gate per chat, and a daily "Purge" evicts anyone who no longer qualifies. Wallet ownership is proven by self-transfer of a small native amount — no WalletConnect, no SIWE, no signed nonce.

03FLOWS

Two swimlanes.

The owner installs the gate. The user proves a wallet and gets in. Both flows are entirely inside Telegram — there is no web application to visit, by design.

Admin onboarding

10 STEPS · DM · OWNER-ONLY
  1. Create gated chatPrivate Group, Channel, or Channel + Discussion. Telegram app, not web.
  2. Create public entry channelSeparate channel where verification CTA lives. Must be public.
  3. Add bot as admin to all chatsDefault permissions; needs can_invite_users + can_restrict_members.
  4. Convert to supergroupToggle history-visible or slow-mode. Side-effect: bot loses admin → re-add.
  5. DM bot /setupPick chat type → pick chat → pick entry channel.
  6. Pay one-time SOL feeAmount and recipient address are not documented.
  7. Pick gate typeToken Gate or Net Worth Gate.
  8. Pick chain + tokenNative, custom contract, or multichain via multichainmetrics.com.
  9. Set threshold + entry messageToken amount or USD value; markdown-style entry copy.
  10. Confirm + activateStored to Settings; bot begins watching.

User verify + join

SELF-TRANSFER · DAILY RE-CHECK
  1. Discover entry channelUser finds the public channel and taps verify CTA.
  2. Open DM with botLikely a t.me/<bot>?start=<token> deep-link binding the user to the chat session.
  3. Pick chain + supply wallet addressOne row per (telegram_user, chain, address) inserted as pending.
  4. Receive tiny self-transfer amountBot generates a unique dust value; user signs nothing — just sends to themselves.
  5. Bot watches the chainTx watcher matches (from == to == address, value == amount) within TTL.
  6. Holdings evaluationBot reads native + token balances across linked wallets, runs gate rules.
  7. Granted accessInvite link issued; bot tracks user against (chat_id, user_id).
  8. Daily PurgeEvery chat re-evaluates. Failures → banChatMember + unbanChatMember. GIF posted.

07V0 PLAN

If we rebuilt it.

Four weeks, single dev. Ship the spine first; defer the differentiators until the foundation is real.

A non-custodial Telegram gating bot in aiogram + Postgres + Redis, with a Mini App for settings, SIWE for EVM and ed25519 for Solana — the spine, then NFT and TON as feature flags.

  1. Bot spine + token gatesaiogram 3 long-poll · /setup wizard · ERC-20 + SPL only · single-wallet per user.
  2. Wallet auth + chain readersSIWE (EIP-4361, 5-min TTL) for EVM · sign-message for Solana · Alchemy multi-chain + Helius. Cache (Redis 60s) on (chain, address, contract).
  3. Daily purge + admin managerAPScheduler · token-bucket pacer (30 msg/sec) · banChatMember + unbanChatMember idempotent.
  4. Mini App settings + shipTelegram WebApp panel · initData HMAC auth · per-chat gates CRUD · whitelist by tg_user_id.
